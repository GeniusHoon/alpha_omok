# [결과보고서] 오목 게임 개발 결과보고서

본 결과보고서는 교내 오목 AI 대전 규칙을 준수하고 고성능의 플레이를 제공하기 위해 진행된 `alpha_omok` 프로젝트 분석, 핵심 알고리즘 설계 및 구현, C++ 포팅을 통한 성능 개선, 그리드 서치 파라미터 튜닝, 그리고 최종 GUI 개발 과정의 모든 성과를 정리하여 명세합니다.

## 1. 개발 계획, 개발 환경 및 일정

### A. 개발 목표 및 사양
* **19x19 오목판 지원**: 표준 19x19 바둑판 규격을 사용하며, X/Y 축 각각 1~19 범위의 좌표계를 정의합니다.
* **대전 규칙의 완벽 준수**: 흑돌(선수)의 중앙 `(10,10)` 자동 착수 강제, 무작위 장애물 돌 3개 배치 규칙, 쌍삼(3-3) 금지 및 장목(5목 이상) 승리 판정을 구현합니다.
* **시간 제한 준수**: AI 연산 및 착수 결정 시간을 **최대 3초 이내**로 설계합니다. (하드웨어 여유를 위해 2.8초 시한 적용)
* **인터페이스 다양화**: 콘솔 기반 텍스트 인터페이스(`play_console.py`)와 고품질 Tkinter GUI 인터페이스(`play_gui.py`)를 모두 제공합니다.

### B. 개발 환경 (Development Environment)
* **운영체제(OS)**: Windows 10 / 11 (x64)
* **개발 언어**: Python 3.8+ / C++ (C++17 표준 지원)
* **컴파일러**: LLVM Clang 17.0+ (`clang++` 활용)
* **핵심 라이브러리**: NumPy, Tkinter (Python 내장 GUI), ctypes (C++ DLL 연동)
* **버전 관리**: Git 및 GitHub

### C. 개발 일정 및 수행 결과
프로젝트는 설계부터 릴리즈까지 총 5단계에 걸쳐 일정 지연 없이 안정적으로 완수되었습니다.
1. **설계 및 분석**: 교내 오목 룰 파악 및 딥러닝 기반 탐색의 병목 분석을 통한 전략 수정 (100% 완료)
2. **콘솔 환경 및 규칙 구현**: 19x19 변환, 장애물 설정, 쌍삼(3-3) 정밀 검사 알고리즘 구현 (100% 완료)
3. **MCTS 알고리즘 및 시간 차단**: 2.8초 스톱워치 연동 Heuristic-MCTS 알고리즘 완성 (100% 완료)
4. **C++ 포팅 및 메모리 최적화**: 핫 패스 반복문 및 트리를 C++ DLL로 마이그레이션하여 성능 100배 향상 (100% 완료)
5. **그리드 서치 및 GUI 릴리즈**: 리그전 그리드 서치를 통한 최적 파라미터 튜닝 및 Tkinter 3D 그래픽 GUI 게임 릴리즈 (100% 완료)

## 2. 오목 게임 분석 및 설계 계획 (주요 코드, 알고리즘 설명)

### A. Inha Omok의 특별한 게임 룰
본 프로젝트에 반영된 교내 오목 대전의 차별화된 룰셋은 다음과 같습니다.
1. **(1,1) 좌하단 원점 좌표계**: 행렬의 좌상단 `(0,0)` 기준 대신 일반 평면 좌표계와 동일하게 작동하며, 내부 행렬 인덱스와 매핑을 위해 아래 공식을 사용합니다.
   * $\text{Row} = 19 - Y$, $\text{Col} = X - 1$
2. **흑돌 선공 핸디캡 (10,10 자동 착수)**: 첫 수가 항상 중앙으로 고정되며 곧바로 백돌의 턴으로 진행됩니다. (Action index = 180)
3. **3개의 장애물 돌 배치**: 게임판 시작 시 무작위(또는 지정) 3개 좌표에 빨간색 장애물이 놓이며, 이 구역은 게임 끝날 때까지 흑/백 모두 착수가 불가능합니다.
4. **쌍삼(3-3) 착수 제한**: 안막힌 3이 동시에 2개 이상 만들어지는 곳은 착수가 금지됩니다. (단, 5목 이상이 동시에 열리는 경우 승리 판정이 우선되어 허용)

### B. 기존 프로젝트의 한계 및 MCTS 결정 배경
* **기존 프로젝트의 구조**: 원래 저장소의 `AlphaOmok`는 알파제로(AlphaZero) 아키텍처를 따라 **MCTS와 딥러닝(ResNet 신경망)을 결합**하여 상태 가치와 정책 확률을 학습시키는 형태였습니다.
* **실전 도입의 한계**:
  1. **막대한 학습 시간 및 리소스**: 19x19 크기의 보드에서 의미 있는 기력의 신경망을 학습시키기 위해서는 고성능 GPU 서버에서 수일에서 수주 간의 자가 대전(Self-play) 학습이 필수적입니다.
  2. **무작위 장애물 룰의 가변성**: 매 대국마다 무작위로 놓이는 3개의 빨간 돌(장애물)은 보드의 대칭성을 깨뜨리고 탐색 공간을 완전히 다른 형태로 바꿉니다. 사전 학습된 신경망은 매번 달라지는 장애물 조합에 유연하게 대처하기 어렵습니다.
* **솔루션 - 휴리스틱 MCTS (Heuristic-MCTS)**:
  * 학습 없이도 강력한 기력을 낼 수 있도록 오목 고유의 패턴(5목, 열린 4, 3, 2목 등)을 정량화한 **휴리스틱 평가 함수**를 설계했습니다.
  * MCTS의 상태 가치 평가($V(s)$)와 사전 확률 정책($P(s,a)$) 계산에 딥러닝 대신 이 휴리스틱 스코어를 결합하여, **학습 자원이 불필요하며 장애물 배치 변화에도 실시간으로 최선의 대응 수**를 읽어내는 하이브리드 탐색 모델을 완성했습니다.

### C. 핵심 코드 및 알고리즘 설명

#### ① 정밀 쌍삼(3-3) 검증 알고리즘 (`check_double_three`)
착수 후보지 $P$에 돌을 놓았을 때, 4개 방향(가로, 세로, 두 대각선)으로 각각 6칸 크기의 윈도우 스캐닝을 진행하여 "열린 3(Active Three)" 패턴의 개수를 집계합니다. 양 끝이 막히지 않은 아래 4개 패턴만 열린 3으로 검증합니다:
* `[0, 0, M, M, M, 0]` (착수 위치가 연속한 3개 돌 중 하나여야 함)
* `[0, M, 0, M, M, 0]`
* `[0, M, M, 0, M, 0]`
* `[0, M, M, M, 0, 0]`

#### ② MCTS 탐색 알고리즘
MCTS는 다음 4단계 주기를 2.8초 시간 제한 내에서 수천 회 반복하여 최적의 수의 확률 분포($\pi$)를 출력합니다.
* **Selection (선택)**: PUCT 공식 $Q(s, a) + U(s, a)$가 최대인 가지를 선택하여 리프 노드에 도달합니다.
* **Expansion & Evaluation (확장 및 평가)**: 리프 노드에서 착수 가능한 모든 행동을 확장하고, 휴리스틱 보드 평가 함수를 통해 상태 가치 $V(s)$를 반환합니다.
* **Backup (역전파)**: 가치 $V(s)$를 루트 노드까지 거슬러 올라가며 방문 횟수($N$)와 가치 합($W$)을 업데이트합니다.

### D. 성능 개선을 위한 C++ 포팅
파이썬 환경에서는 19x19 대형 보드 상의 무수한 패턴 매칭과 수천 번의 노드 확장 연산을 3초 내에 수행하는 데 큰 성능적 병목이 발생했습니다. 이에 따라 탐색 엔진의 모든 핵심 로직을 **C++(`omok_cpp.cpp`)**로 포팅하고 ctypes로 연동해 극한의 최적화를 이뤄냈습니다.

#### 주요 최적화 기법
1. **Zero-Allocation 노드 풀(Node Pool) 도입**: MCTS 노드 확장 시 `new`/`delete` 힙 메모리 할당 지연과 단편화를 막기 위해 BSS 영역에 `2,000,000` 크기의 정적 전역 노드 풀 배열을 사용했습니다.
2. **정적 스택 배열 대체**: 자주 연산되는 로컬 스코어링 함수 내에서 동적 벡터(`std::vector`)를 제거하고 고정 크기의 정적 스택 배열과 포인터 연산으로 전면 대체하여 메모리 오버헤드를 제로화했습니다.
3. **32바이트 구조체 캐시 얼라인먼트**: `MCTSNode` 구조체를 정확히 32바이트로 정렬(`alignas(32)`)하고 데이터형을 압축하여 CPU 캐시 미스를 방지했습니다.
4. **결과**: 초당 MCTS 탐색 횟수를 **평균 1,785 SPS(Simulations Per Second)**로 대폭 높여, 3초 이내에 5,000회 이상의 깊은 수읽기를 완벽히 수행하게 되었습니다.

### E. 파라미터 튜닝을 위한 그리드 서치 (Grid Search)
AI의 탐색 효율성과 공수 균형을 튜닝하기 위해 핵심 하이퍼파라미터의 격자 탐색(Grid Search) 툴을 개발하여 최적 조합을 도출했습니다.

* **튜닝 파라미터**:
  - `C_puct` (MCTS 탐색 탐험 가중치): `[1.0, 3.0, 5.0, 7.0]`
  - `Defense Weight` (상대 위협 차단 수비 가중치): `[1.0, 1.2, 1.5]`
* **검증 구조**: 총 12가지 조합이 서로 대국을 펼치는 라운드 로빈 리그전(총 66쌍 × 대칭 4판 = 264판)을 진행했습니다.
* **최적의 조합 도출**:
  - **`C_puct` = 3.0, `Defense Weight` = 1.2** 조합이 리그전에서 가장 높은 승률을 기록하여 최종 기본 매개변수로 탑재되었습니다.
  - 진행 상황을 실시간으로 ETA 및 매치 정보와 함께 파악할 수 있도록 갱신 속도를 최적화했습니다.

### F. GUI 개발
사용자 경험을 극대화하기 위해 Tkinter 라이브러리를 사용해 고급스러운 대국용 GUI 프로그램(`play_gui.py`)을 구축했습니다.

* **비동기 멀티스레딩 구현**: MCTS가 연산하는 동안 GUI 이벤트 루프가 굳어 창이 멈추는 현상을 방지하고자, MCTS를 백그라운드 스레드로 실행하고 GUI 메인 스레드에서 `root.after`를 통해 상태를 주기적으로 감시 및 제어하는 구조를 확립했습니다.
* **입체적인 3D 그래픽**: 광원 처리를 시뮬레이션한 하이라이트 레이어와 드롭 섀도우(Drop Shadow)를 중첩 렌더링하여 실물 조약돌 같은 미려한 바둑돌 디자인을 구현했습니다.
* **편의 기능**: 
  * 마우스 호버 시 착수할 위치의 예비 돌을 가이드라인(점선 원)으로 사전 표시.
  * 마지막 착수지 노란색 링 표시.
  * 쌍삼 금지 영역 클릭 시 빨간색 주의 알림 레이블 출력.
  * 게임 히스토리 스택을 활용한 AI/유저 동시 2수 무르기(Undo) 및 보드 리셋 기능.
