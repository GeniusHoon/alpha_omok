## 1. 대전 필수 개발사항 및 콘솔 인터페이스 설계

### A. 19x19 바둑판 및 좌표계 변환 설계
- **요구사항**: X축(1~19), Y축(1~19)을 사용하며, 일반적인 XY평면의 원점(1,1)이 바둑판의 좌측 하단(Bottom-Left)에 위치함.
- **분석**: 파이썬의 NumPy 행렬은 `[row][col]` 구조를 가지며, 좌측 상단(Top-Left)이 `(0,0)`입니다. 따라서 다음과 같은 양방향 변환 공식이 요구됩니다.
  
  $$\text{Column (col)} = X - 1$$
  $$\text{Row (row)} = 19 - Y$$

  - **예시**: 좌하단 원점 $(1,1)$ $\rightarrow$ `row = 18`, `col = 0`
  - **예시**: 우상단 $(19,19)$ $\rightarrow$ `row = 0`, `col = 18`
  - **예시**: 중앙점 $(10,10)$ $\rightarrow$ `row = 9`, `col = 9`

- **구현 설계**: `utils.py` 또는 `env_competition.py` 내에 좌표 변환 헬퍼 함수 구현:
  ```python
  def coord_to_matrix(x, y):
      return 19 - y, x - 1

  def matrix_to_coord(row, col):
      return col + 1, 19 - row
  ```

### B. 유저 콘솔 입력 인터페이스 설계
- **요구사항**: 유저가 좌표를 `x,y` 형태(예: `10,10` 또는 `3,15`)로 콘솔에 직접 입력하여 착수할 수 있도록 함.
- **입력 형식 및 예외 처리**:
  1. 사용자로부터 한 줄을 읽어와 쉼표(`,`)를 기준으로 분리합니다: `input_str.split(',')`
  2. 각각 정수형 변환이 가능한지 검사합니다: `1 <= x <= 19` 이고 `1 <= y <= 19` 인지 확인.
  3. 이미 돌이 놓여진 자리이거나, 빨간색 장애물 돌이 배치된 자리이거나, 안막힌 3-3 자리가 아닌지 검증(Validation)합니다.
  4. 입력이 잘못된 경우(형식 에러, 범위 초과, 금지수 등), 패널티 패배를 방지하기 위해 콘솔 창에 경고 및 재입력 유도 메시지를 출력합니다.
- **콘솔 바둑판 출력(Rendering)**:
  - 19x19 바둑판의 상태를 콘솔에 텍스트 형태로 아름답게 렌더링합니다.
  - 가로/세로 축 번호(1~19)를 바둑판 테두리에 출력하여 유저가 좌표를 쉽게 파악할 수 있도록 합니다.
  - 흑돌은 `●` (혹은 `O`), 백돌은 `○` (혹은 `X`), 빨간색 장애물은 `■` (혹은 `R`), 빈 공간은 `.` 또는 `+` 로 표시합니다.

### C. 실시간 연산 및 시간 제한 (3초 이내) 설계
- **요구사항**: 인공지능이 계산하여 착수하기까지의 연산 시간이 최대 3초 이내여야 함.
- **구현 설계**: `agents.py`의 MCTS 탐색 루프에 실시간 시간 측정 코드를 연동합니다.
  ```python
  def _mcts(self, root_id):
      start_time = time.time()
      for i in range(self.num_mcts):
          # 2.8초 경과 시 강제 종료하여 0.2초의 하드웨어 마진 확보
          if time.time() - start_time > 2.8:
              break
          
          leaf_id, win_index = self._selection(root_id)
          value, reward = self._expansion_evaluation(leaf_id, win_index)
          self._backup(leaf_id, value, reward)
  ```

### D. 흑돌(선수) 우위 보완을 위한 규칙 설계
1. **중앙 착수 강제 (10,10)**:
   - 게임 시작 시 흑돌이 선택되면 첫 착수 행동을 취하지 않고 자동으로 `(10, 10)` (인덱스 `[9, 9]`)에 흑돌을 배치하며, 턴을 백돌(Turn 1)로 넘깁니다.
   - 첫 턴 `root_id`는 자동으로 `(0, 180)`으로 초기화됩니다. (19x19 판에서 `[9,9]` 인덱스는 `9 * 19 + 9 = 180`에 해당)
2. **빨간색 장애물 돌 (3개) 배치**:
   - 게임 시작 전 콘솔 입력 프롬프트를 통해 3개의 장애물 좌표(예: `x1,y1 x2,y2 x3,y3`)를 입력받아 배치합니다.
   - 보드 상에서 빨간 돌은 상수 `2`로 표기되며, 콘솔 렌더링 시 `■` 또는 빨간색 텍스트로 강조합니다.
   - 이 3개의 좌표는 흑돌과 백돌 모두 놓을 수 없는 장애물 지역으로 설정합니다.
   - `legal_actions` 함수에서 해당 좌표들을 제외하도록 설계합니다.
     ```python
     def legal_actions(node_id, board_size, obstacles):
         all_action = set(range(board_size**2))
         placed = set(node_id[1:])
         obstacle_indices = set(obstacles)
         actions = all_action - placed - obstacle_indices
         return list(actions)
     ```

### E. 안막힌 3,3 (쌍삼) 금지 규칙 설계
- **요구사항**: 안막힌 3,3은 둘 수 없으며, 한 쪽이 막힌 4,3이나 5,3 등은 허용됩니다.
- **판정 알고리즘 설계**:
  1. 가상 착수: 빈 칸 중 임의의 위치 $P$에 현재 플레이어의 돌을 임시로 놓아봅니다.
  2. 4가지 방향(가로, 세로, 우상 대각선, 우하 대각선)에 대해 패턴을 분석합니다.
  3. 각 방향에서 임시 배치한 돌을 포함하여 생성된 3개 연속 돌이 "열린 3"인지 판별합니다.
     - 양 끝에 최소 1칸 이상의 빈 공간이 있어야 하고, 장애물(빨간 돌)이나 적의 돌에 의해 막히지 않아야 합니다.
  4. '열린 3'의 개수가 2개 이상일 경우 해당 위치로의 착수를 금지합니다.
  5. 단, 장목(5목 이상)이 동시에 완성되는 착수는 금지수 판정보다 승리 판정을 우선시하여 허용합니다.

### F. 승리 판정 (5개 이상) 설계
- **요구사항**: 5개 이상의 흑돌 또는 백돌을 나열하면 승리합니다.
- **구현 설계**: `utils.py`의 `check_win` 함수 내에서 연속된 돌의 개수가 5개 이상(>= 5)인 경우를 찾아 승리로 리턴합니다.

## 2. 휴리스틱 평가 함수 (Heuristic Evaluation Function) 세부 정의

19x19 바둑판의 무작위 장애물 조건 속에서 학습 없이 3초 이내에 고성능 탐색을 가능하게 하기 위해, **패턴 매칭 기반 휴리스틱 평가 함수**를 아래와 같이 정의합니다.

### A. 돌 배치 패턴별 점수 정의
보드 상의 가로, 세로, 양방향 대각선 라인을 스캔하며, 연속된 5~6칸 윈도우 내에서 자신의 돌(`M`), 상대방의 돌/장애물(`O`), 빈 공간(`.`)의 배치를 매칭하여 점수를 부여합니다.

| 패턴 명칭 | 패턴 예시 (자신의 돌: `M`, 적/장애물: `O`, 빈칸: `.`) | 부여 점수 | 설명 |
| :--- | :--- | :---: | :--- |
| **오목 (Five)** | `[M, M, M, M, M]` | **100,000** | 즉시 승리하는 최고의 가치 |
| **열린 4목 (Active Four)** | `[., M, M, M, M, .]` | **10,000** | 다음 턴 절대 승리 보장 (양방향 대기) |
| **닫힌 4목 (Blocked Four)** | `[O, M, M, M, M, .]` / `[., M, M, ., M, M, .]` 등 | **1,000** | 한쪽이 막혔거나 빈칸이 껴있는 4목 |
| **열린 3목 (Active Three)** | `[., ., M, M, M, .]` / `[., M, ., M, M, .]` 등 | **1,000** | 수비하지 않으면 다음 턴 열린 4목이 됨 |
| **닫힌 3목 (Blocked Three)**| `[O, M, M, M, ., .]` / `[., O, M, M, ., M]` 등 | **100** | 한쪽이 막힌 3목 |
| **열린 2목 (Active Two)** | `[., ., M, M, ., .]` / `[., M, ., M, ., .]` 등 | **100** | 공격의 기초가 되는 열린 2목 |
| **닫힌 2목 (Blocked Two)**| `[O, M, M, ., ., .]` 등 | **10** | 한쪽이 막힌 2목 |
| **단일 돌 (Single Stone)**| 주변 빈 공간에 인접한 돌 1개 | **1** | 공격 전개를 위한 기본 점수 |

---

### B. 상태 가치 평가 함수 ($V(s)$) 계산 공식
보드 상태 $s$에서 리프 노드(Leaf Node)의 가치 $V(s)$는 다음과 같이 계산됩니다.

1. 보드의 모든 좌표를 기준으로 4대 방향(가로, 세로, 우상 대각선, 우하 대각선)에 대해 윈도우 스캐닝을 진행합니다.
2. 현재 턴 플레이어(Self) 입장에서 매칭된 모든 패턴의 점수 합산 ($Score_{\text{self}}$)을 계산합니다.
3. 상대방(Opponent) 입장에서 매칭된 모든 패턴의 점수 합산 ($Score_{\text{opp}}$)을 계산합니다.
4. 두 점수의 차이(Net Score)를 구한 뒤, 이를 신경망의 가치 출력 범위인 $[-1.0, 1.0]$ 사이로 정규화하기 위해 **하이퍼볼릭 탄젠트(tanh)** 함수를 사용합니다.

$$V(s) = \tanh\left(\frac{Score_{\text{self}} - Score_{\text{opp}}}{C}\right)$$

*여기서 정규화 상수 $C$는 $10,000$으로 설정합니다. 이는 하나의 열린 4목(10,000) 또는 여러 개의 열린 3목 차이가 날 경우 평가값 $V(s)$를 $1.0$ (승리 확정) 또는 $-1.0$ (패배 확정)에 가깝게 포화(Saturate)시키기 위함입니다.*

---

### C. 탐색 우선순위 정책 함수 ($P(s, a)$) 계산 공식
MCTS의 확장(Expansion) 단계에서 임의의 후보 착수점 $a$에 대해 탐색할 우선순위 사전 확률(Prior Probability) $P(s, a)$를 다음과 같이 결정합니다.

1. 후보 착수점 $a$에 돌을 놓았을 때 얻을 수 있는 가상의 보드 스코어 변화량($\Delta Score$)을 계산합니다.
   - **공격 가치($A_a$)**: 착수 후 자신의 점수 증가량.
   - **수비 가치($D_a$)**: 해당 자리에 상대방이 두었을 때 얻었을 상대방 점수 증가량(방어 효과).
2. 행동 점수 $Score(a)$를 정의합니다. 상대방의 결정적인 수(열린 3, 4목 등)를 차단하는 수비적 행동에 가중치를 두기 위해 수비 가중치 $\beta = 1.2$를 곱해줍니다.
   
   $$Score(a) = A_a + \beta \cdot D_a$$

3. 이를 거듭제곱 분포 스케일링(Power Scaling)을 통해 정규화하여 모든 합이 1.0이 되도록 확률 분포로 만듭니다. (수학적으로 로그 소프트맥스 $\text{Softmax}(\ln(Score(a) + 1.0) / \tau)$와 동일하며, 지수 오버플로우/언더플로우를 방지하고 초반과 종반 탐색을 유연하게 연결합니다)
   
   $$P(s, a) = \frac{(Score(a) + 1.0)^{1/\tau}}{\sum_{b \in \text{LegalActions}} (Score(b) + 1.0)^{1/\tau}}$$

*여기서 $\tau$는 탐색의 다양성을 제어하는 온도 매개변수(Temperature, 기본값 2.0)입니다. 이를 통해 AI가 가장 점수가 높은 공격/방어 포인트 위주로 MCTS Tree의 가지치기를 매우 좁고 깊게 수행(Selection)하여, 3초 이내에 수읽기 깊이를 극대화할 수 있습니다.*
